## 問題

README 裡有一個結論是：T-MAC 用 µ=4(16 格表)所以能用 `shuffle_epi8` 查表，LUT-GEMM 官方用 µ=8(256 格表)超過 shuffle 能定址的範圍，只能用 `gather`，所以慢很多。

那麼我希望驗證，那我就把這個表變小，是不是就可以解決問題了？

另外如果因此不用用到 gather，是不是也可以加速？

### 什麼時候能用 shuffle，什麼時候不能

shuffle 有兩個硬性限制：

1. 表只能有 16 格。index 是 4-bit，只能選 0~15
2. shuffle 查的是 byte(int8)，不是 float。原始的 LUT 值如果是 float32，沒辦法直接查，要先量化成 int8 才能用。

### 換成 shuffle 的 tradeoff

- float 查表值要轉成 int8，這本身要花時間算 scale、四捨五入，而且真實資料（在我的測試剛好是整數）量化通常會有誤差，準確度會打折扣。
- int8 的範圍只有 -128~127，查表結果加一加很容易超出範圍（overflow），所以中間要先轉成 int16、甚至 int32 才能安全累加，多了幾道轉換的手續，程式也變複雜。
- 因為 index 只能是 4bit，兩個 index 通常要塞進同一個 byte才不會浪費空間，原本的資料佈局要重改。

## 做法

拿官方 LUT-GEMM(µ=8、gather)當基準，改出兩個版本：

1. 只縮小 µ，指令一樣：µ 從 8 改成 4(表從 256 格變 16 格)，但查表還是用 gather。
2. µ 縮小，指令也換掉：µ=4，改用 shuffle_epi8，做法比照 T-MAC。

測資跟正確性檢查方式都跟原本一樣，跑 1000 次取平均。

## 結果

GFLOPS:

| K | 基準(µ=8， gather) | 只縮小µ(µ=4， gather) | µ=4 + shuffle |
|---:|---:|---:|---:|
| 256 | 4.53 | 2.54 | 22.41 |
| 1024 | 5.29 | 2.64 | 42.41 |
| 2048 | 5.36 | 2.65 | 28.10 |
| 4096 | 4.13 | 2.46 | 16.33 |
| 8192 | 3.64 | 2.14 | 6.03 |

perf 抓到的數字：

| 版本 | IPC | cache miss 率 |
|---|---:|---:|
| 基準 | 0.16 | 28.0% |
| 只縮小µ | 0.174 | 22.3% |
| µ=4+shuffle | 0.485 | 59.1% |

## 可看到的

### 只縮小表格不換指令，更慢

IPC 沒什麼變化，但總指令數幾乎翻倍——因為表變小，要分的組數變多(K/8 變 K/4)，gather 要多呼叫一倍次數。這說明 gather 慢是它本身的固定成本，跟表多大沒關係，表變小只是害你多呼叫幾次而已。

### 換成 shuffle 之後，中小規模加速許多

K=1024 快了 8 倍(42.41 -> 5.29)，IPC 也從 0.16 跳到 0.485。這證實了查表指令改掉確實是關鍵。

不過 K=8192並沒有跟著變快，實際的原因還不知道。（investigating/TODO）

## 結論

1. 要設計的是怎麼樣能夠用shuffle來加速。而表必須符合一定的處理，否則就只能退回去用 gather
2. µ 本身不是關鍵，單純把表縮小又不換指令，反而更慢。
3. 一旦表格小到能用 shuffle，效能可以差好幾倍
